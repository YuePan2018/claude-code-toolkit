# 项目设计

把一个模糊想法收敛成清晰的路线选择和可执行的实施计划。

## 工作原则

- 先调研，再设计，不要默认必须自研。
- 先看完整产品和可复用项目，再看库、框架和底层组件。
- 对价格、维护状态、版本变化、API 能力、平台规则等时效性信息，优先使用最新来源验证。
- 使用中英文双语关键词搜索，避免只看到单侧资料。
- 默认用中文输出，保留关键技术术语的英文原文。
- 默认在对话中给出结论和方案；仅在用户明确要求时再落成文档或代码。

## 阶段 1：调研现有方案

先判断应该买、用、fork，还是自研。

### 搜索范围

- 商业产品与 SaaS
- GitHub 上可运行的完整开源应用
- 可直接集成的 MCP 服务、API、SDK、模板和 boilerplate
- Product Hunt、AlternativeTo、G2 等目录或评测站点

### 输出内容

- 已有方案清单：名称、链接、定位、核心能力、定价或许可证、活跃度
- 可复用候选：最接近需求的产品或仓库，以及复用方式
- 差异化空间：现有方案覆盖不到的需求、体验缺口或集成缺口

## 阶段 2：给出路线决策

根据调研结果明确建议，不要模棱两可。

- 如果成熟产品已经满足需求，直接建议使用现成方案，并说明原因与代价
- 如果开源项目接近需求，建议基于现有项目二次开发，并列出需要补齐的部分
- 如果现有方案不足以满足目标，说明为什么值得自研，然后进入后续设计

把结论写成一句清晰建议，例如：`建议先 fork X，再补齐 A/B/C；暂不建议从零开发。`

## 阶段 3：拆解项目设计

仅在自研或深度改造确实合理时进入这一阶段。

### 先明确

- 目标用户是谁
- 核心问题是什么
- 成功标准是什么
- MVP 必须包含什么，哪些可以延后

### 再做模块拆解

为每个模块说明：

- 职责
- 主要输入与输出
- 依赖的上游与下游模块
- 关键风险或技术难点
- 是否已有可复用组件

同时给出：

- 推荐技术栈及理由
- 重要权衡：开发速度、维护成本、性能、扩展性、部署复杂度
- MVP 优先级排序

在继续细化实现前，先让用户确认模块划分与技术路线。

## 阶段 4：规划实现路径

在用户确认方向后，再生成详细实施方案。

为每个确认模块补充：

- 可复用资源：库、框架、参考仓库、MCP 工具、API 服务
- 实现设计：目录结构、核心对象或函数、数据流、接口边界
- 集成点：认证、存储、同步、权限、通知、观测、部署
- 测试策略：单元测试、集成测试、手工验证重点
- 里程碑：按阶段拆分为可交付的步骤

在输出详细方案前，再次检查时效性较强的信息，特别是：

- 定价或免费额度
- 维护状态和最近更新
- 重大版本变化或破坏性升级
- 新出现的更优替代方案

## 推荐输出结构

按下面的顺序组织回复，必要时可合并：

1. 项目理解与假设
2. 现有方案调研
3. 路线建议：使用 / fork / 自研
4. 模块设计
5. 技术选型与权衡
6. 实施里程碑
7. 仍需用户确认的问题

## 质量门槛

- 不要为了显得完整而堆砌技术名词；结论必须服务决策
- 不要把库列表当方案；要说明为什么选
- 不要忽略成本、交付速度、维护复杂度、合规性等非功能需求
- 不要在调研不足时假装确定；明确标注假设和不确定性
- 如果更适合直接购买或采用现成工具，要明确说出来
